GTID를 이용한 Mariadb Replication

개요

  1. 클라이언트가 마스터노드에게 쓰기 트랜잭션을 요청한다
  2. 바이너리로그가 업데이트된다 (바이너리로그에는 디비의 이벤트도 포함된다)
  3. 슬레이브가 마스터로부터 정보를 전달받는다
  4. 슬레이브는 이 정보를 릴레이로그에 첨부한다
  5. 슬레이브 SQL 스레이드는 릴레이로그에 있는 명령문을 수행반복한다

복제메커니즘

  1. 표준복제: 각 노드는 자신만의 트랜잭션 ID를 갖는다
  2. 글로벌 트랜잭션(GTID, Global Transaction ID): 모든 노드는 같은 트랜잭션 ID를 갖는다 (MariaDB 10부터 가능)

환경

vm002: master
vm003: slave

마스터설정

GTID를 사용하기 위해 gtid_strict_mode를 활성해야함
활성화시키위해서 서버를 재시작해야 한다

[mysqld]
server-id = 1
character-set-server = utf8
bind-address = 0.0.0.0
log_bin = /var/lib/mysql/mariadb-bin #빈로그 저장경로
expire_logs_days = 2 #빈로그가 무작성 커지는것을 방지하기위해 2일이 지나면 플러시한다
sync_binlog = 0 #1로 설정하면 빈로그파일을 디스크 동기화한다. 복제성능을 높이려면 0으로
slave_compressed_protocol = 1 #압축을 사용하면 cpu를 더많이 사용하지만 빨라진다
binlog_format = row #row, statement, mixed가 있다, row가 제일 안전하다
gtid_strict_mode = 1

마스터에 허용할 슬레이브계정을 만든다

create user 'replication@vm003' identified by 'password';
grant replication slave on *.* to 'replication@vm003';
flush privileges;

슬레이브설정

[mysqld]
server-id = 2
character-set-server = utf8
bind-address = 0.0.0.0
slave_compressed_protocol = 1 #마스터,슬레이드 둘다 설정되어야함
binlog_format = row
read_only
gtid_strict_mode = 1

마스터에서 DB에 변화를 주면 이벤트가 발생하고 아래와 같이 이벤트 번호를 확인가능하다

슬레이브에서는 마스터의 gtid를 확인을 하려면 먼저 슬레이브 노드를 재시작한다

master_use_gtid에 사용가능한 옵션은 slave_pos, current_pos, no가 있다

MariaDB [(none)]> stop slave;
MariaDB [(none)]> reset slave;
MariaDB [(none)]> select global slave_pos;
MariaDB [(none)]> set global slave_pos="0-1-1"; #마스터노드와 동일하게 설정
MariaDB [(none)]> change master to master_host='vm002', master_user='replication', master_password='password', master_use_gtid=slave_pos;
MariaDB [(none)]> start slave;

슬레이브 상태확인

MariaDB [(none)]> show slave status\G;
...
 Slave_IO_Running: Yes
 Slave_SQL_Running: Yes
...
